# 共通

すべてのレビューで確認する。プロジェクト固有の境界は [project.md](project.md) を優先する。

## 正しさ

- [ ] 変更の意図が diff から読み取れる
- [ ] 正常系・異常系・境界値（空配列、nil/null、0 件、未認証など）を考慮している
- [ ] エラー処理が握りつぶされていない（ログも出ない silent failure がない）
- [ ] 既存の呼び出し元・契約（API shape、job 引数、型）を壊していない

## セキュリティ

- [ ] SQL インジェクション、XSS、CSRF、パス traversal の余地がない
- [ ] 認証・認可の漏れがない（未ログイン・権限不足で到達できる操作がない）
- [ ] 機密情報（トークン、パスワード、内部 URL）のハードコードやログ出力がない
- [ ] ユーザー入力をそのまま HTML / SQL / シェルに渡していない

## パフォーマンス

- [ ] N+1 クエリ、不要な全件取得、ループ内 I/O がない
- [ ] 同期処理に重い処理（外部 API、大量 DB、ファイル I/O）を載せすぎていない
- [ ] キャッシュや再検証の粒度が妥当（過剰 invalidate / 無限 refetch がない）

## 可読性・保守性

- [ ] 命名が意図を表している（禁止サフィックス `Manager` / `Util` / `Service` 等の曖昧名を増やしていない）
- [ ] 関数・クラスの責務が 1 つに収まっている
- [ ] デメテルの法則: 不要な連鎖呼び出し（train wreck）がない
- [ ] コメントは非自明な業務ルール・制約だけに留まっている
- [ ] 同じ意図のコードを字面だけで共通化していない（意図が違う重複は許容）

## テスト

- [ ] 変更に見合うテストがある（新規ロジック・バグ修正・リグレッションしやすい箇所）
- [ ] テストが実装の詳細ではなく振る舞いを検証している
- [ ] テスト名・配置が既存パターンと揃っている

## スコープ

- [ ] 依頼範囲を超える変更（無関係な整形、ついでリファクタ）が混ざっていない
- [ ] YAGNI: 将来用の抽象化・設定・拡張ポイントを先に作っていない
